# Задача 3: Технологична архитектура — Дигитален процес по жалби

Този документ описва високо ниво архитектура за услугата „Дигитален процес по жалби“ в UniCredit Bulbank. Източникът на истината е диаграмата [`architecture.drawio`](architecture.drawio), разработена в [diagrams.net](https://app.diagrams.net). Архитектурата е организирана в **7 зони** с **две защитни стени** (периметрена и вътрешна), съгласно стандартния банков модел за сегментация.

## Как се чете диаграмата

Диаграмата се чете **отгоре надолу**, по ред на **нарастващо доверие** спрямо банковата инфраструктура:

1. Client / Internet — хора и клиентски канали
2. Периметрена защитна стена
3. DMZ и външни партньори — крайно терминиране
4. Вътрешна защитна стена
5. Приложна зона — API Gateway, Orchestrator, AI Copilot, IdP
6. Зона на данните — PostgreSQL, Redis, Object storage, Audit log
7. Core banking — CBS / CRM / CARD_SYSTEM (наследени)

## Зона 1 — Клиент (публичен Интернет)

**Клиент** — автентикиран потребител на банката; самоличността е доказана чрез логина (биометрия / ПИН / OIDC), без нужда от КЕП.

**Bulbank Mobile / Online** — основният канал за подаване на жалби. Използва **TLS 1.2+** и **OIDC** към банковия IdP.

**Push delivery (device)** — APNs / FCM; push нотификациите се задействат **от банковия сървър** чрез Push API, а не директно от клиентското приложение.

## Зона 2 — Периметрена защитна стена

**Perimeter firewall (L3/L4)** — разрешава само входящ **TCP 443 (HTTPS)** от Интернет към ръба на DMZ. Трафикът се терминира на reverse proxy / WAF, не директно на вътрешни приложни сървъри.

## Зона 3 — DMZ (Linux периметър)

**Reverse proxy + WAF** — **NGINX + ModSecurity** върху **RHEL 9**, публичен слушател **443/tcp HTTPS**. Функции: TLS терминиране, WAF правила срещу OWASP top 10, маршрутизиране към вътрешния API Gateway.

## Зона 4 — Външни доверени партньори (egress-controlled)

- **Push providers (APNs / FCM)** — HTTPS 443, използвани за нотификации към клиента.
- **Email / SMS gateway** — SMTP TLS 587 / HTTPS 443, за дублиране на потвърждения извън приложението (необходимо при регулаторни изисквания за писмена комуникация).
- **Regulators (БНБ / КЗП / Помирителна комисия)** — периодично регулаторно отчитане чрез HTTPS + SFTP. Не е realtime връзка.

Изходящият трафик към тези хостове минава през **allowlist на защитна стена**, **прокси** и мониторинг.

## Зона 5 — Вътрешна защитна стена

Сегментирани правила; приложната зона приема **mTLS само от DMZ reverse proxy**, а не произволни клиенти.

## Зона 6 — Приложна зона

### API Gateway

- **Продукт:** Spring Cloud Gateway
- **Среда:** Java 21 с Tomcat 10 (вграден application server)
- **Порт / канал:** вътрешен HTTPS 8443 / mTLS
- **Роля:** централен вход за API, маршрутизация, лимити, куки за автентикация, корелационни идентификатори

### Complaints Orchestrator (банков сървър)

- **Съответствие:** изрично връзван с „Банков сървър“ от sequence диаграмата
- **Стек:** Spring Boot върху JBoss EAP 8
- **Интерфейси:** REST/JSON, OIDC client към IdP, async callbacks за дълги процеси
- **Роля:** реализира процеса — валидации, състояния (чернова → регистрирана → разследване → решение → затворена), маршрутизация към специалист, интеграция с CBS/CRM

### AI Copilot service

- **Стек:** Python (FastAPI), self-hosted, вътрешен REST — никакви клиентски данни не напускат банковата мрежа
- **Функции:** асемблиране на контекст от CBS / CRM / transaction history и изготвяне на чернова на отговор за специалиста
- **Регулаторна роля:** AI **не** взема решения и не изпълнява автономно монетарни или регулирани действия (съгласно EBA/GL/2015/18) — специалистът винаги приема или коригира предложението преди то да се изпрати към клиента.

### Bank IdP / IAM

- **Продукт:** Keycloak (пример)
- **Протоколи:** OIDC/OAuth2 на 443 — издаване/валидиране на токени за мобилен вход и API модели

## Зона 7 — Зона на данните (чувствителни / висока увереност)

| Компонент | Технология | Порт / протокол | Роля |
|---|---|---|---|
| **Complaints DB** | PostgreSQL 16 · RHEL 9 | 5432/tcp (TLS) | Система на записа за жалби: статуси, референтни номера, история, решения |
| **Redis** | Redis | 6379/tcp (TLS) | Сесии и идемпотентност (защита от двойно подаване, безопасни повтори) |
| **Object Storage** | MinIO / S3-compatible | HTTPS 443 | Съхранение на прикачени файлове (сканирани документи, скрийншоти) |
| **Audit log** | Append-only (WORM) | TLS | Регулаторна проследимост — всяко действие по случая; изисква се за EBA, ЗПУПС и DORA |

Само приложният слой достига тези услуги; **няма директен Интернет достъп**.

## Зона 8 — Core banking (CBS + CRM + CARD_SYSTEM)

Система на запис за клиенти, сметки, карти, транзакции. Complaints Orchestrator **не замества** CBS, а се интегрира чрез **REST/JSON + mTLS + allowlist** за:

- извличане на контекст (сметки, транзакции, история)
- изпълнение на монетарни действия (връщане на такса, компенсация)
- обновяване на CRM случаи за omnichannel видимост

## Таблица „стрелка → протокол“

| От | До | Протокол | Стил |
|---|---|---|---|
| Клиент → Mobile/Web | — | uses | тънка |
| Mobile/Web → Reverse proxy | HTTPS / 443 | плътна |
| Reverse proxy → API Gateway | HTTPS (mTLS to app tier) | плътна |
| API Gateway → Orchestrator | REST / JSON | плътна |
| API Gateway → IdP | OIDC | пунктирана |
| Orchestrator → AI Copilot | REST / JSON (internal) | плътна |
| Orchestrator → PostgreSQL | JDBC (TLS) | плътна |
| Orchestrator → Redis | Redis TLS | плътна |
| Orchestrator → Object Storage | S3 API (HTTPS) | плътна |
| Orchestrator → Audit log | append (TLS) | плътна |
| Orchestrator → CBS | REST / JSON (mTLS) | плътна |
| Orchestrator → Push providers | Push API | пунктирана |
| Orchestrator → Email/SMS | SMTP TLS / HTTPS | пунктирана |
| Orchestrator → Regulators | HTTPS + SFTP | пунктирана |

**Правило:** плътно = банков контролиран път; пунктирано = партньорски изход или странична плоскост (OIDC).
